# MemTest 生成器问题诊断与优化方向

> **区分**：
> - **generator.py** = 程序化生成（自己造数据，零依赖）
> - **knowledge_builder.py** = 语料输入生成（从文本提取，需要 LLM）

---

## 一、程序化生成（generator.py）的问题

| 问题 | 严重程度 | 说明 |
|------|---------|------|
| **数据池太小** | 🔴 高 | 城市10个、人名8个、地点7个、产品7个 → 大量重复 |
| **版本文本过短** | 🔴 高 | 平均37字符，像填表而非自然叙述 |
| **重复查询** | 🟡 中 | 30条查询中有4条重复（模板化生成） |
| **聚类单一** | 🟡 中 | 只生成1个cluster，无法测试聚类召回 |
| **推理链缺失** | 🟡 中 | reasoning_chain=0，无链式数据 |
| **无负样本查询** | 🔴 高 | 所有查询都有正确答案，无法测试 Precision |
| **时间戳精度** | 🟢 低 | 所有记忆都是当前时间回溯，无真实时间跨度感 |
| **事件类型单一** | 🟢 低 | 只有交易/会议/日常三类，缺乏情感/冲突/发现等 |
| **versions 3种差异不够** | 🟢 低 | 标准/详细/口语的区分度不明显，像同一模板的变体 |

---

## 二、语料输入生成（knowledge_builder.py）的问题

| 问题 | 严重程度 | 说明 |
|------|---------|------|
| **LLM 事实提取格式不稳定** | 🔴 高 | 要求"每行一个JSON对象"，但LLM常输出 `[...]` 或带markdown |
| **版本生成质量差** | 🔴 高 | v2/v3 只是模板拼接，不是真正的不同表达风格 |
| **查询生成依赖程序化模板** | 🟡 中 | 从事实生成的查询与 generator.py 一样模板化 |
| **缺少跨语言支持** | 🟡 中 | 英文文本的处理不够健壮 |
| **链构建过于简单** | 🟡 中 | 按人物简单分组，未考虑因果关系或时序 |
| **无负样本查询** | 🟡 中 | 所有查询都有正确答案，无法测试 Precision |
| **字段标准化粗糙** | 🟢 低 | location 简单 split，未处理"北京 vs 北京市" |
| **无质量校验** | 🟡 中 | 生成后没有自动检查环节 |

---

## 三、提示词（prompts/）的问题

| 问题 | 严重程度 | 说明 |
|------|---------|------|
| **prompts 没有 few-shot 示例** | 🟡 中 | LLM 容易输出格式不一致 |
| **query_generate 提示词缺少"难度"判断标准** | 🟡 中 | 简单/中等/困难的定义模糊 |
| **无 chain 生成提示词** | 🟡 中 | knowledge_builder 中的链构建是代码硬编码 |
| **无负样本生成提示词** | 🟡 中 | 测试 precision 需要 |
| **fact_extract 的 dynasty 字段在英文文本中无用** | 🟢 低 | 对 HP/英文小说 benchmark 不适用 |

## 四、程序化生成优化方向（generator.py）

### 方向1：扩大数据池 + 提升多样性

```python
# 当前：CITIES = 10个，应扩展到 50-100 个
# 当前：人名8个，应扩展到 200+ 个（含英文）
# 新增事件类型：情感、冲突、发现、转折、意外、灾难等

# 建议：按"领域"分数据池
DOMAINS = {
    "business": {"cities": [...], "people": [...], "events": [...]},
    "historical": {"dynasties": [...], "figures": [...], "events": [...]},
    "scifi": {"planets": [...], "characters": [...], "events": [...]},
    "daily": {"cities": [...], "people": [...], "events": [...]},
}
```

### 方向2：真正的语义多样化版本

当前：3 个版本只是模板拼接，长度/用词差异不大。

优化：让 versions 之间有**真正的语义差异**：
- v1 标准：客观描述，含时间地点人物
- v2 主观：从人物视角，含心理活动/评价
- v3 间接：第三方转述，含推测/省略细节

### 方向3：负样本查询（关键缺失）

当前：所有查询都有预期答案，无法测试 false positive。

优化：生成两类负样本：
1. **相关但无关**：人物正确，但事件/地点错误（如"张伟在星巴克购买茅台"→实际在肯德基）
2. **完全不相关**：人物/事件/地点都不在数据库中

### 方向4：链式数据生成（推理测试核心）

当前：reasoning_chain=0，无链式数据。

优化：显式构建因果链：
```json
{
  "memory_id": "MEM001",
  "reasoning_chain": "CHAIN_001",
  "chain_position": 1,
  "cause": "",
  "effect": "MEM002"
}
```

需要新增 `chain_generate` 提示词，让 LLM 从文本中提取因果序列。

### 方向5：聚类数据生成（整理测试）

当前：只生成1个cluster。

优化：显式构建多主题聚类：
- 同一人物的多个事件 → person_cluster
- 同一地点的多个事件 → location_cluster
- 同一产品/项目的多个事件 → topic_cluster
- 同一时间段的多个事件 → temporal_cluster

每个 cluster 3-8 条记忆，确保有合理的聚类测试数据。

## 五、语料输入生成优化方向（knowledge_builder.py）

### 方向6：LLM 提取稳定性

- 改用 `generate_json()` 方法（已提供），自动处理 markdown 包裹
- 批量提取：每篇文章一个 LLM call → 改为每 BATCH 篇文章一个 call
- 失败重试：3 次失败回退到简单提取模式

### 方向7：版本生成质量

- 用 LLM 生成真正的 3 种表达风格（而非模板拼接）
- 或至少让模板拼接更自然：加入转折词、省略次要信息

### 方向8：查询生成质量

- 用 LLM 从事实生成多样化查询（而非模板化）
- 生成负样本查询（与某记忆相关但不对）

### 方向9：链构建改进

- 不只看人物，看**因果连接词**："因为...所以..."、"导致..."、"随后..."
- 用 LLM 判断事件间是否有因果关系
- 链长度 3-6 跳，覆盖简单/中等/困难

### 方向10：性能优化

```python
# 阶段2：每个字段一个 call → 改为批量分类（30个词/次）
# 阶段6：每个查询一个 call → 改为批量预解析（15个/次）
# 利用 llm_interface.py 的 batch_generate() 并行
```

## 六、提示词优化方向（prompts/）

| 提示词 | 优化 |
|--------|------|
| `memory_enhance` | 加入 few-shot 示例，明确输出格式，增加"主观视角"和"间接转述"版本 |
| `query_generate` | 加入负样本生成指令，明确难度判断标准，加 few-shot |
| `fact_extract` | 支持中英文，加入 "chain_id" 提取，让 LLM 标注事件间的因果关系 |
| 新增 `chain_extract` | 从文本提取事件序列，标注 cause→effect 关系 |
| 新增 `negative_query` | 生成与某记忆"相关但不对"的查询，用于测试 precision |

---

## 七、通用优化：数据质量校验

新增 `quality_check.py`（生成后运行）：
- 检查所有 expected_memory_ids 是否有效
- 检查查询是否有重复文本
- 检查版本文本长度分布是否合理
- 检查类别/难度分布是否均衡
- 输出质量报告，不达标可自动重试

## 八、优先级建议

### 程序化生成（P0-P2）

| 优先级 | 任务 | 预期收益 |
|--------|------|---------|
| P0 | 负样本查询生成 | 可测 precision，当前无法测 |
| P0 | 扩大数据池（200+人名/50+城市） | 减少重复，提升 realism |
| P1 | 链式数据生成（chain_extract） | 推理测试有数据可测 |
| P1 | 多聚类生成（4-6个cluster） | 整理测试有数据可测 |
| P2 | 版本语义差异化 | 检索 recall 更真实 |
| P2 | 质量校验脚本 | 自动发现问题 |

### 语料输入生成（P1-P3）

| 优先级 | 任务 | 预期收益 |
|--------|------|---------|
| P1 | 提示词加 few-shot | 提升LLM输出稳定性 |
| P1 | 批量LLM调用优化 | 生成速度提升3-5x |
| P2 | 链构建改进（因果关系） | 推理链更真实 |
| P2 | 负样本查询 | 可测 precision |
| P3 | 跨语言支持完善 | 英文 benchmark 更自然 |
| P3 | 按领域分数据池 | 支持更多 benchmark 场景 |
