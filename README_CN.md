# MemTest

> AI 记忆系统通用评测工具包。接入任意记忆系统，输出完整评测报告。

[English](README.md)

---

## 为什么需要 MemTest？

AI Agent 越来越依赖长期记忆——但你怎么知道你的记忆系统是否真的*好用*？大多数团队用临时脚本做检索评测，测试数据和后端强耦合。MemTest 把它们解耦：

- **一次编写，处处评测** — 同一套测试可跑在任意记忆系统上
- **6 大评测维度** — 不只看召回率，还看存储完整性、聚类、遗忘、推理和检索深度
- **零依赖** — 纯 Python 标准库 + JSON，无框架锁定，无安装地狱
- **合成或真实数据** — 程序化生成万级测试用例，或从自有语料构建

## 快速开始

```python
from runner import MemoryTestSuite, MemoryAdapter, load_test_db

# 1. 为你的记忆系统实现 3 个方法
class MyAdapter(MemoryAdapter):
    def reset(self):
        self.db.clear()

    def store(self, memory_text: str, metadata: dict):
        self.db.insert(text=memory_text, **metadata)

    def search(self, query: str, top_k: int = 20) -> list[dict]:
        return self.db.query(query, limit=top_k)

# 2. 加载评测数据 & 跑评测
db = load_test_db("sample_db_100.json")
suite = MemoryTestSuite(MyAdapter())
report = suite.run(db)
print(report.summary())
```

## 评测维度详解

### 1. 存储完整性（Storage Integrity）

**设计目的**：验证记忆系统能否完整、准确地写入所有记忆条目。这是最基本的能力——如果存都存不进去，后面的检索都是空谈。

**测试内容**：
- 按简单/中等/困难三级生成记忆，困难级包含更多元数据字段
- 每条记忆包含 3 种风格版本（标准叙述、详细描述、口语化），系统需全部存储
- 最终检查 `存储数 / 总条目数`，理想值为 300%（每条记忆存 3 个版本）

**广度**：覆盖 6 种事件类型（交易/会议/决策/日常/技术/情感），100+ 城市，70+ 地点类型

### 2. 检索 Precision/Recall（Retrieval）

**设计目的**：核心维度——给定一条查询，系统能否找回对应的正确记忆？这是记忆系统的命脉。

**测试内容**：5 种查询类型，从单一维度到多维组合逐级递增难度：

| 查询类型 | 示例 | 难度递进 |
|---------|------|---------|
| **人物检索** | "魏强最近做了什么" | 单维度：人名匹配 |
| **地点检索** | "在银川会议室B发生过什么" | 单维度：地点匹配 |
| **事件检索** | "关于茅台的事件有哪些" | 单维度：事件/产品匹配 |
| **时间检索** | "查询2周前在太原的相关记录" | 单维度：**时间匹配，含相对时间计算** |
| **组合检索** | "1周前卢杰在健身房的技术事件" | 三维度：时间+人物+地点交叉约束 |

**时间检索的特殊设计**：每条记忆包含 4 种时间表示：
- `absolute`：精确时间戳（`2026-03-15 21:21:12`）
- `relative`：相对时间（`2个月前`、`3天前`、`1周前`）
- `fuzzy`：模糊时间（`上个月`、`前几天`、`很久以前`）
- `timestamp`：Unix 时间戳

查询使用 `relative`/`fuzzy` 表述，但记忆中存的是 `absolute`。这意味着系统必须理解"2周前"对应哪个时间区间，才能正确召回——这不是简单的字符串匹配。

**难度分级**：每条记忆标注简单/中等/困难，困难级包含更多干扰项和更模糊的查询表述。

### 3. 整理聚类（Organization/Clustering）

**设计目的**：语义相关的记忆是否被系统识别为"同一组"？比如同一个人在不同时间的多笔交易，系统检索到其中一条时，是否也能关联到同组的其他记忆？

**测试内容**：
- 每 10 条记忆归入一个 `cluster_id`（如 `CLUSTER0001`），同组记忆共享事件类型
- 用组内一条记忆的关键词（人名+产品）检索，检查 Top-10 结果中是否包含同组其他成员
- `cluster_accuracy` = 命中同组的查询数 / 总查询数

**广度**：每组 10 条，覆盖所有事件类型

### 4. 遗忘合理性（Forgetting）

**设计目的**：人类记忆有"用进废退"——高频访问的记忆应被优先保留，低频/偶发的记忆可以逐渐淡出。一个好的 AI 记忆系统也应体现这一特性。

**测试内容**：
- 每条记忆标注 4 种衰减级别：`高频记忆`、`中等频率`、`低频记忆`、`偶发事件`
- 附带 `access_count`（0-100 的访问次数），模拟记忆的使用频率
- 用记忆原文检索，分别统计高频组和低频组的保留率（retention）
- `forgetting_ratio_valid = 高频保留率 > 低频保留率`，即系统是否正确地把高频记忆排在前面

**评判标准**：不是要求系统永不遗忘，而是要求遗忘有方向性——重要记忆优先保留。

### 5. 逻辑推理（Reasoning）

**设计目的**：真实场景中，用户的问题往往需要跨多条记忆推理。比如"张伟和王芳一起做了什么"需要同时命中两个人的记忆并关联。系统能否支持多跳查询？

**测试内容**：
- **因果/时序/对比/包含/推导** 5 种推理类型
- `组合检索` 查询：多维度交叉约束（时间+人物+地点）
- `组合推理` 查询：需要跨多条记忆链式推理
- `logic_accuracy`：逻辑查询的命中率
- `chain_accuracy`：链式推理的完整命中率

### 6. 深度检索（Deep Retrieval）

**设计目的**：长期记忆的真正挑战不是找回"昨天的事"，而是找回"半年前只提过一次的事"。随着时间推移和语义距离增大，检索难度指数级上升。系统能否在不同深度层级保持召回能力？

**测试内容**：
- 所有记忆的时间跨度拉到 1 年~2 年前（而非其他维度的近期记忆）
- 每条记忆标注：
  - `layers`（3-7）：概念层级深度
  - `associations`（2-5）：关联数量
  - `semantic_distance`：`near`/`mid`/`far`，语义距离
- 分别统计近/中/远三个层级的召回率
- 理想系统：近层接近 100%，远层仍有合理召回

## 评测结果

以下结果基于我们自研的 **NOESIS-II** 记忆系统，使用四大名著数据集（11,794 条记忆，8,907 条查询）的 500 条随机采样，Top-20 召回：

| 方法 | 整体 Recall | 水浒传 | 西游记 | 三国演义 | 红楼梦 |
|------|------------|--------|--------|---------|--------|
| jieba + SQL LIKE | 2.2% | 0.9% | 1.0% | 2.2% | 4.5% |
| TF-IDF + 余弦相似度 | 53.0% | 85.1% | 53.1% | 47.8% | 28.2% |
| Sentence-Transformers | *待测* | - | - | - | - |

**评测方法说明**：
- **jieba + SQL LIKE**：对查询做 jieba 分词，生成多词 OR LIKE 子句，直接在 NOESIS-II 的 SQLite 数据库中 SQL 查询，取 Top-20
- **TF-IDF + 余弦相似度**：对全量记忆和查询做 jieba 分词后构建 TF-IDF 矩阵（sklearn），计算查询与记忆的余弦相似度，阈值 0.3，取 Top-20
- **Sentence-Transformers**：使用 `paraphrase-multilingual-MiniLM-L12-v2` 编码，ChromaDB 向量检索，评测进行中

**关键发现**：
- 关键词匹配（jieba + LIKE）对语义检索基本失效——"袁绍做了什么"搜不到"袁绍引兵往渤海而去"
- TF-IDF 提升了 24 倍，但在半文言文本上仍吃力（红楼梦：28.2%）——jieba 分词对古汉语效果差，语义鸿沟靠词频无法弥合
- 按查询类型：组合推理 100%（关键词多易命中）、人物 55.1%、时间 52.6%、事件 45.7%、地点 29.7%
- 神经向量检索预期可将 Recall 推到 70-80%+，尤其改善红楼梦半文言场景
